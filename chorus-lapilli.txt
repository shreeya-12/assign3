SETUP: same as Tic-Tac-Toe. Also requires Node.js and react.

GAME: After having the Tic-Tac-Toe working, I added the xPieces and oPieces variables
defined with let and assigned to squares.filter(s => s === "#").length, where # is the 
letter of the current player. Then I removed squares[i] from the handleClick initial
if/else block because a spot being occupied doesn't necessarily stop a move from 
happening in this version of the game. I then added the selected and setSelected states 
to keep track of the selected squares for a player's move after 3 pieces from each have 
been placed on the board. Then I added a nested if statement in the else block of the 
winner if/else at the end of the Board component, where if (selected) returns true, then 
the phrase " select next destination" is added to the end of the status variable.

Then I worked on an isMovable function to keep track of valid moves since chorus lapilli 
has different rules than Tic-Tac-Toe. It takes four parameters, the squares array, start index,
end index, and the letter of the player currently playing. It checks if end is a valid spot 
to move to, and I used a two-dimensional array similar to lines in calculateWinner since it 
would be easier to compare valid spots that way: validIndices[start] gets the valid indexes 
for that position in the board. If validIndices[start] includes end and the center square has 
a piece of the current player and the start index is not the center square, then the function 
creates a new copy of the squares parameter called newSquares and does the move, testing it with 
returning calculateWinner(newSquares) as the result. If the condition doesn't work, then the 
function returns null.

Then I added the logic for the movement and pieces handling in a large if/else if/else block
in handleClick. The first condition checked if xIsNext and xPieces were less than 3. If true, 
then it would check if squares[i] had a value and if that was true the function would return.
Otherwise, nextSquares[i] would be set to "X". The else if block after this else block would 
check the opposite, where !xIsNext and oPieces < 3 should return true, with the same inner logic 
except for "X" being replaced with "O".

The next else if block checked if xIsNext and xPieces === 3 return true. If so, then there is 
a nested if/else block. If (selected) was not null, then there is another nested if/else block. 
If isMovable returns a non-null value then nextSquares[i] was set to the letter and nextSquares[selected]
was set to null. Then xIsNext is set to !xIsNext, selected is set to null, squares is set to nextSquares,
and the function returns. Else, setSelected(null) is called and the function returns. If selected did
not return a truthy value, then the else condition for the parent if/else calls setSelected(i) and the
function returns. The final else if block checked if !xIsNext and oPieces === 3, and if so, the same 
thing happens, except instead of X it's with O. The exactly same logic continues, but isMovable is 
called with "O" as the final parameter.

If handleClick reaches the end of the if/else without returning, then set squares to nextSquares 
and set xIsNext to !xIsNext.

An issue that arises from this code is that nothing happens, which I realized was because isMovable 
returned null as default. Instead I made isMovable return validIndices[start].includes(end) because the 
if statement only checked for the move including the center condition, so returning this fixed it
because a piece could move and the game would go on.

This introduced me to another issue because sometimes a player could make invalid moves and 
place a fourth piece and override another player's move. So, I added an if(squares[end]) return false;
statement to beginning of the isMovable function because occupied spots cannot be changed. This fixed 
the issue and pieces are moving only if they can. However, a player can click on the opposite player's 
letter and move that to another spot, replacing it with their own piece. To fix this, change the first 
two lines in the isMovable if statements in the logic
        nextSquares[i] = "O"; OR nextSquares[i] = "X";
        nextSquares[selected] = null;
to the following:
        nextSquares[i] = squares[selected];
        nextSquares[selected] = squares[i];
I made this change because I was trying to swap the values, and reassigning them might have been an issue.

The next change was in the else conditions for if (selected) in both else if blocks in handleClick.
I changed it to 
        if (squares[i] === "X") {
            setSelected(i);
            return;
        }

After these change, extra pieces weren't being added to the board and players' moves were not skipped. This 
also allowed the player to revert to the start of the move if they try to make an invalid move intially.

The only blocking issue now was that any piece in the top left corner was not moving even if the move 
was valid. The only failed test after trying to submit this was that repeated tile swaps have no issues. 

I then changed the center rule test in isMovable to 
        if (calculateWinner(newSquares)) {
            return true;
        } else {
            return false;
        }
instead of just returning calculateWinner(newSquares) because calculateWinner returns player letter or null,
which is the opposite of what isMovable is trying to return. Then, based on a Piazza recommendation, I changed 
all the if (selected) statements to if (selected !== null) which fixed the top left corner issue and the 
autograder issues. The reason this was an issue was because 0 could be considerd falsy, making the previous 
version of the condition not evaluate to true, which could have made the center rule and movement rules 
not be measured properly.

There are three tests in test_chorus_lapilli.py. One to test if a move has been made, one to test if a 
specific winning combination has been made, and one to test if a piece has been moved.
